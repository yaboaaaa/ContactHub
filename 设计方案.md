# 多用户通讯录系统 — 完整设计方案

## 一、项目概述

构建一个支持多用户、内置管理员、具备导入/导出/回收站等完整功能的在线通讯录服务。系统采用 **Spring Boot 3.x + Spring Data JPA + Bootstrap 5** 技术栈，一套代码同时适配 PC 与移动端，部署为内嵌网页的独立 JAR 包。

**核心功能清单**

- 用户注册/登录，基于 Session 的安全认证
- IP 暴力破解防护（同一 IP 连续失败 3 次锁定 5 分钟）
- 可选验证码机制（easy-captcha 图形验证码）
- 系统内置唯一管理员账号，可管理所有用户（增、删、改、禁用/启用、重置密码）
- 分组管理（默认分组受保护，删除分组时联系人自动归入默认分组，事务保证）
- 联系人信息全面（姓/名拆分 CardDAV 兼容、性别、三类电话、邮箱、单位、职位、省/市/区、详细地址、生日、备注）
- 多功能组合查询：JPA Specification 动态查询，按姓名、电话、地域、单位、职位、分组等筛选
- 头像上传（Thumbnailator 缩放 200×200，存入服务器文件系统）
- Excel 导入/导出通讯录（Apache POI，17 字段，三种导出模式）
- 回收站（软删除，支持恢复、彻底删除、清空）
- 国际化中英文切换（200+ 翻译条目）
- Swagger/OpenAPI 接口文档
- Docker 容器化部署 + GitHub Actions CI/CD
- 全局输入校验（Bean Validation）、权限隔离、事务保障、全局异常处理

## 三、用户角色与权限

| 角色 | 权限 |
| :--- | :--- |
| **ADMIN** | 唯一内置管理员（账号在 application.yml 中配置，应用启动时自动创建）。可查看所有用户及其联系人、管理用户（增删改禁用/启用/重置密码）、操作回收站（查看/恢复/删除所有用户联系人） |
| **USER** | 通过公开注册获得。只能管理自己的分组和联系人，使用查询、导入导出、回收站等功能。不能查看其他用户的数据 |

权限控制通过 `@PreAuthorize("hasRole('ADMIN')")` 注解限制管理员接口，Service 层通过 `SecurityContextHolder` 获取当前用户 ID 手动校验 `user_id` 实现数据隔离。

## 六、核心业务设计

### 6.1 用户注册与管理员初始化

- **管理员初始化**：在 `application.yml` 中配置默认管理员用户名和密码，应用启动时通过 `AdminInitializer`（实现 `CommandLineRunner`）检查是否存在该账号，若不存在则自动创建，密码 BCrypt 加密，角色设为 `ADMIN`，且不可被删除或降级。同时为管理员创建默认分组。
- **用户注册**：公开注册表单，前端异步校验用户名唯一性、实时检测密码强度（弱/中/良/强四级）。提交后后端校验用户名格式（英文开头，4-20 位字母数字）、密码强度（至少 6 位）、邮箱格式。密码 BCrypt 加密，角色固定为 `USER`，同时自动为其创建一个"默认分组"（`is_default=true`），使用 `@Transactional` 保证原子性。
- **管理员管理用户**：管理员可在 `/admin/users` 页面查看所有用户，执行禁用/启用、修改信息（不能修改角色为 ADMIN）、重置密码、删除用户（物理删除其所有分组、联系人及回收站数据）。

### 6.2 分组管理

- 每个用户登录后可见自己的分组列表（默认分组排首位，按 `sort_order` + `created_at` 排序）。
- 新增分组时校验同名冲突（同一用户名下不允许重复分组名称）。
- 编辑分组：修改分组名称，默认分组（`is_default=true`）的名称不可修改。
- 删除分组：**非默认分组**才可删除。删除前将该分组下所有未被软删除的联系人（`is_deleted=false`）的 `group_id` 更新为当前用户的默认分组 ID，然后删除分组。整个过程在一个事务中完成，若发现是默认分组则抛出业务异常。
- 默认分组不可删除，前端隐藏其删除按钮，后端二次校验。

### 6.3 联系人 CRUD 与软删除

- 新增联系人：必须提供姓名和至少一个联系电话（手机/家庭/工作至少填一个），若未指定分组则归入默认分组。Contact 实体在 `@PrePersist` 时自动生成 UUID 作为 `uid` 字段（CardDAV 兼容），`name` 字段由 `family_name + given_name` 自动拼接。
- 编辑联系人：校验归属权（当前用户或管理员），支持更换所属分组。
- 删除操作：**前端"删除"执行软删除**，即将 `is_deleted` 置为 `true`，记录 `deleted_at`；在分组删除时仅移动未删除联系人。
- 回收站：
  - 查看：`GET /api/v1/contacts/recycle/data?page=&size=`，仅展示 `is_deleted=true` 的联系人，使用 `@EntityGraph` 预加载 group 关联。
  - 恢复：`PUT /api/v1/contacts/{id}/restore`，将 `is_deleted` 设为 `false`，清除 `deleted_at`。
  - 彻底删除：`DELETE /api/v1/contacts/{id}/permanent`，物理删除记录。
  - 清空回收站：`DELETE /api/v1/contacts/recycle`，物理删除当前用户所有已删除联系人。

### 6.4 多条件组合查询

采用 **JPA Specification** 动态组合查询条件，代码清晰且无需引入 MyBatis。

查询接口 `GET /api/v1/contacts/data` 接受参数：

- `keyword`：模糊匹配姓名（LIKE，转义 `%`、`_` 通配符防注入）
- `phone`：模糊匹配手机、家庭、工作电话（三个字段 OR 组合）
- `province`, `city`, `district`：精确匹配
- `company`：模糊匹配
- `jobTitle`：模糊匹配
- `groupId`：按分组筛选
- `page`, `size`：分页

Service 层构建 Specification，对字符串使用 `%value%`（并转义 `%`、`_` 等通配符），所有查询默认添加 `user_id = :currentUserId AND is_deleted = false`（回收站除外）。使用 `root.fetch("group", JoinType.LEFT)` 预加载分组关联，避免 `LazyInitializationException`。

### 6.5 头像上传与存储

- 上传接口：`POST /user/avatar`（用户头像）、`POST /api/v1/contacts/{id}/avatar`（联系人头像），接收 `MultipartFile`。
- 校验：仅允许 `image/jpeg`、`image/png`，大小不超过 2MB，扩展名白名单校验。
- 处理：使用 Thumbnailator 将图片缩放为 **200×200 像素** 的正方形（中心裁剪），输出质量 80%，转为字节数组存入 `uploads/avatars/` 目录，数据库记录 `avatar_url` 路径。旧头像文件自动删除避免磁盘空间浪费。
- 读取：`GET /user/avatar` 或 `GET /user/{id}/avatar`，根据 `avatar_url` 返回图片，设置正确的 `Content-Type`。
- **CDN 预留**：当未来需要迁移至 CDN 时，可编写批处理将本地文件上传并更新 `avatar_url`，代码中仅需调整存储后端。
- **首字母头像**：未上传头像的联系人可通过 `InitialsAvatarUtil` 自动生成姓名首字母头像。

### 6.6 导入导出通讯录

**导出 Excel**

- 接口：`GET /api/v1/contacts/export`（按筛选条件）、`GET /api/v1/contacts/export/all`（全部）、`POST /api/v1/contacts/export/selected`（选中 ID 列表）。
- 使用 Apache POI 的 `XSSFWorkbook`，表头 17 列：姓、名、显示名、性别、手机、家庭电话、工作电话、邮箱、单位、职位、省、市、区、详细地址、生日、备注、所属分组。
- 性别转为文字（男/女/不便透露），分组显示名称（通过关联查询）。
- 响应头设置 `Content-Disposition: attachment; filename="contacts.xlsx"`。

**导入 Excel**

- 接口：`POST /contacts/import`，接收上传的 Excel 文件。
- 逐行解析，每行校验：
  - 姓名必填，否则跳过。
  - 分组处理：如果填写了分组名，先查找该用户下是否存在，不存在则自动创建；未填写则归入默认分组。
  - 数据格式校验：手机号正则（可选）、邮箱格式、日期格式等。
  - 根据姓名+手机号判断重复（默认跳过重复记录）。
- 记录成功/失败行及原因，返回成功条数和失败详情列表。
- 限制文件大小（10MB）和最大行数（5000 行），防止 OOM。

### 6.7 校验与鲁棒性

- **Bean Validation**：所有 DTO 标注 `@NotBlank`、`@Size`、`@Email`、`@Pattern` 等注解。ContactDTO 中手机号使用正则 `^(\d{6,20})?$` 校验，各字段限制最大长度。
- **全局异常处理**：`@ControllerAdvice` 的 `GlobalExceptionHandler` 捕获 `MethodArgumentNotValidException`、`BusinessException`、`EntityNotFoundException` 等，返回统一的 `ApiResult` 格式 JSON 响应。
- **统一 API 响应格式**：所有 REST 接口返回 `ApiResult<T>`（code、message、data 三个字段），通过静态工厂方法 `success()`、`success(data)`、`error(code, message)` 简化响应构造。
- **事务**：涉及多表操作（注册、分组删除、导入）均使用 `@Transactional`。
- **权限隔离**：Service 层通过 `SecurityContextHolder` 获取当前用户，所有查询均附加 `user_id` 条件；管理员可通过管理员接口跨用户操作。
- **防 XSS/注入**：前端纯静态 HTML + AJAX 获取 JSON 数据（无服务端模板渲染），JPA 参数化查询，LIKE 查询转义通配符。
- **IP 暴力破解防护**：`IpLockFilter`（OncePerRequestFilter）在 Spring Security 认证链之前拦截登录请求，`LoginAttemptService` 基于 `ConcurrentHashMap` 记录失败次数，3 次失败锁定 5 分钟。

- 个人信息页面（user/profile.html）：头像展示和上传，密码修改

全部使用 Bootstrap 5 栅格与组件，确保手机触屏友好。

## 八、安全设计

- 使用 Spring Security 的 `HttpSecurity` 配置：
  - 登录页、注册页、静态资源、Swagger、i18n 接口允许匿名访问。
  - 其他所有请求需认证。
  - 启用 CSRF，使用 `CookieCsrfTokenRepository`（HttpOnly=false），AJAX 请求从 meta 标签读取 token。
- 密码编码器：`BCryptPasswordEncoder`。
- IP 暴力破解防护：`IpLockFilter` 在认证前检查，`LoginAttemptService` 基于 ConcurrentHashMap 记录失败次数，3 次失败锁定 5 分钟。
- 可选验证码：`CaptchaService` 生成和校验图形验证码，可通过 `captcha.enabled` 配置开关。
- 管理员账号初始化后禁止注册同名账号（数据库唯一约束）。
- Session 超时 30 分钟，超时后重定向登录页。
- 方法级权限控制：`@EnableMethodSecurity(prePostEnabled = true)` + `@PreAuthorize("hasRole('ADMIN')")`。
- 全局异常处理：`GlobalExceptionHandler` 统一捕获异常，返回友好错误信息。

## 九、国际化设计

- 后端：Spring MessageSource 加载 `messages_zh.properties`（中文）和 `messages_en.properties`（英文），共 200+ 条翻译条目。
- 前端：通过 `/api/v1/i18n?lang=zh` 接口获取翻译文本 Map，JavaScript 函数 `t(key, fallback)` 实现运行时文本查找。
- HTML 元素通过 `data-i18n` 属性标记需要翻译的文本内容，`data-i18n-placeholder` 标记占位符文本。
- 用户语言偏好存储在 `localStorage` 的 `site_lang` 字段中，切换语言后页面自动刷新重新加载翻译。

## 十、部署与运行

**本地运行**

```bash
# 创建数据库（可选，JPA 会自动建表）
mysql -u root -p -e "CREATE DATABASE contacthub DEFAULT CHARACTER SET utf8mb4;"

# 配置数据库连接（编辑 application-local.yml 中的数据库连接信息）

# 构建并运行
mvn clean package -DskipTests
java -jar target/addressbook.jar
```

**Docker 部署**

```bash
# 构建镜像
docker build -f docker/Dockerfile -t contacthub:latest .

# 运行容器
docker run -d -p 8087:8087 \
  -e MYSQL_HOST=your-mysql-host \
  -e MYSQL_USER=root \
  -e MYSQL_PASSWORD=your-password \
  --name contacthub contacthub:latest
```

**GitHub Actions CI/CD**

推送到 main/master 分支时自动触发：运行测试 → 构建 Docker 镜像（标签：latest + Git 短哈希）→ 推送到阿里云 ACR。

需要在 GitHub Secrets 中配置：`ACR_KEY`、`ACR_SECRET`、`ACR_NAMESPACE`。
